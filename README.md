# 隐私邮件搜索与解密系统

一个安全且隐私优先的邮件搜索系统，允许用户在**不向服务器泄露邮件内容或搜索关键词**的前提下，完成邮件的搜索与解密，基于 Flask、Jieba（中文 NLP）和现代 Web 加密技术构建。

## 数据集

项目使用的数据集 TREC 2006 Spam Corpus (TREC06c)，由清华大学提供，包含约 50,000 封真实邮件。由于是早期数据，文件编码混杂（主要是 GBK/GB2312，少量 UTF-8），包含中文、英文、HTML标签、附件头信息等噪声，这里将非垃圾邮件的 21,766 封邮件作为数据集。
数据预处理在 client_se_index_builder.py 中实现：
鲁棒读取 (Robust Reading)：采用 errors='ignore' 策略，并尝试 gbk -> gb18030 -> utf-8 多重编码回退。关键词提取时使用工具 jieba 进行中文分词，正则移除所有非中文字符，过滤停用词，对每封邮件提取 Top 50 高频词作为索引指纹。
分块 (Bucketization)：将 21,766 个文件按顺序切分为 22 个块 (Block)，前 21 块每块 1000 个文件，最后一块 766 个，这样做是为了实现 PIR，把全库搜索缩小为块内搜索来降低计算量，同时分块可以方便数据集扩充。

## 数据集加密

系统采用了 “信封加密” 模式，

数据层：每个邮件文件使用 AES-256-GCM (对称加密) 加密。
密钥层：每个 Block 生成一个随机的 AES 块密钥。
封装层：使用用户的 RSA-2048 公钥 加密 AES 块密钥。

这样的情况下，加密过程既有对称加密的速度，又有非对称加密的安全性，同时服务器只有公钥，无法解密数据。

## 系统中的使用者

1.  **数据管理员 (Data Admin)**：负责数据的“生产”和“加密上架”，拥有原始明文，但**不持有**用户私钥。
2.  **密钥管理员 (Key Admin)**：负责公钥基础设施 (PKI) 管理，验证用户身份并注册公钥，**不接触**明文数据。
3.  **普通用户 (End User)**：只持有私钥，负责“解密下载”，**不接触**原始明文文件。
4.  **服务器 (Server)**：纯粹的“盲存储”和“盲计算”节点，存储密文和哈希索引，**零知识**。


| 角色 | 核心职责 | 持有密钥 | 可见数据 | 不可见数据 |
| :--- | :--- | :--- | :--- | :--- |
| **数据管理员** | 1. 清洗原始邮件2. 构建 SE 索引 (Hash)3. 分块加密文件 (AES+RSA)4. 上传密文与索引 | **无** (或临时持有用于加密的随机密钥) | 原始明文邮件 | 用户的搜索内容、用户的解密行为 |
| **密钥管理员** | 1. 生成/接收用户公钥2. 将公钥绑定到用户身份3. 维护用户白名单 | **无** (存储用户公钥) | 用户公钥、用户ID | 邮件内容、搜索关键词、用户私钥 |
| **普通用户** | 1. 本地保存私钥2. 搜索邮件 (发送 Hash)3. 本地解密查看 | **私钥 (Private Key)** | 解密后的目标邮件 | 其他用户的邮件、服务器上的其他密文 |
| **服务器** | 1. 存储加密文件块2. 存储哈希索引3. 响应搜索与下载请求 | **无** (只有系统运行权限) | 密文文件、哈希值、文件元数据 | **所有明文内容**、**搜索语义**、**用户身份与行为的关联** |

## 业务流程

### 系统初始化与数据上架 (数据管理员操作)

此阶段由**数据管理员**在离线或受信任环境中完成，完成后将产物部署到服务器。

1.  **准备数据**：
    *   获取原始邮件数据集。
    *   运行 `client_se_index_builder.py` 。
2.  **构建索引与加密**：
    *   **SE 索引构建**：提取关键词 $\to$ 计算 $H(keyword)$ $\to$ 映射到文件 ID。
    *   **混合加密**：
        *   生成随机 AES 密钥 $K_{block}$。
        *   用 $K_{block}$ 加密该块内所有邮件 $\to$ `file.enc`。
        *   此时**不需要**特定用户的公钥。数据管理员使用一个**系统主公钥 (System Master Public Key)** 或者预留加密位。
            *   **代理重加密/多收件人**：数据管理员使用**自己的公钥**加密块密钥。当新用户加入时，密钥管理员使用新用户的公钥对块密钥进行“重加密”或“二次封装”。
                1.  数据管理员生成数据密文和 **未加密的 (或系统公钥加密的) 块密钥列表**。
                2.  上传到服务器。
                3.  **密钥管理员**在用户注册时，获取该用户的公钥，从服务器拉取块密钥列表，用**用户公钥**重新加密块密钥，写回服务器的 `manifest.json` 中对应用户的字段。
                
        1.  数据管理员生成数据密文 (AES)。
        2.  数据管理员生成每个块的 AES 密钥 $K_{block}$。
        3.  数据管理员**不直接加密 $K_{block}$**，而是将 $K_{block}$ 以**明文形式**（在安全通道）交给**密钥管理员**，或者使用**密钥管理员的公钥**加密后存储。
        4.  数据管理员上传 `Encrypted_Files` 和 `SE_Index`。`Manifest` 中的 `Encrypted_Key` 字段初始为空，或者由**密钥管理员**在用户注册时动态生成。
        
### 用户注册与密钥注入 (密钥管理员操作)

1.  **用户生成密钥对**：
    *   用户在本地运行脚本生成 $(PubKey_{User}, PrivKey_{User})$。
    *   用户保留 $PrivKey_{User}$ 。
    *   用户将 $PubKey_{User}$ 发送给密钥管理员。
2.  **密钥管理员注册**：
    *   登录管理后台，录入用户信息。
    *   上传 $PubKey_{User}$。
    *   **触发密钥封装任务**：
        *   系统读取原始的块密钥 $K_{block}$ (由数据管理员安全提供，或系统内部持有)。
        *   使用 $PubKey_{User}$ 加密 $K_{block}$ $\to$ $EncKey_{User, Block}$。
        *   将 $EncKey_{User, Block}$ 写入该用户的专属配置区。

### 用户搜索与解密 (普通用户操作)

1.  **身份验证**：
    *   用户打开网页，加载本地 $PrivKey_{User}$。
2.  **搜索 (SE)**：
    *   输入关键词 $w$。
    *   客户端计算 $T = H(w)$。
    *   发送 $T$ 到服务器。
    *   服务器匹配 $T$，返回文件列表 `[File_A, File_B]` 及其所在的 `Block_ID`。
3.  **获取密钥 (Key Retrieval)**：
    *   用户点击 `File_A` (位于 `Block_5`)。
    *   客户端请求 `GET /api/user_block_key?block_id=5`。
    *   服务器查找**当前登录用户**对应的 `EncKey_{User, Block_5}` 并返回。
    *   不同用户拿到的 `EncKey` 是不同的，但解密后得到的 $K_{block}$ 是一样的。
4.  **本地解密**：
    *   客户端用 $PrivKey_{User}$ 解密得到 $K_{block}$。
    *   下载 `File_A.enc`。
    *   用 $K_{block}$ 解密文件内容。

## ✨ 核心功能

- 🔒 **端到端加密**：所有邮件使用 AES-256-GCM 算法加密存储
- 🔍 **隐私搜索**：基于加密索引的可搜索加密（SE）技术，服务器无法获知搜索词
- 🇨🇳 **中文 NLP 支持**：集成 Jieba 分词，专为中文邮件内容提取高质量关键词
- 💻 **客户端解密**：所有解密操作均在浏览器本地完成（基于 Web Crypto API），私钥不出域
- 🧩 **模块化设计**：清晰的后端（Flask）与前端（JS）分离架构，易于扩展

## 🛠️ 技术栈

| 组件 | 技术选型 |
|-----------|------------|
| **后端框架** | Python 3.8+, Flask |
| **加密库** | `cryptography` (RSA-OAEP, AES-GCM) |
| **前端 UI** | HTML5, Bootstrap 5, jQuery |
| **前端加密** | Forge.js (RSA), Web Crypto API (AES) |
| **中文处理** | Jieba (中文分词与关键词提取) |
| **数据存储** | 本地文件系统 (分块存储加密数据) |

## 🚀 快速开始

### 1. 环境准备
- Python 3.8 或更高版本

### 2. 安装依赖
```bash
git clone https://github.com/StuBoo3i/PrivateMailSearch-Decrypt.git
pip install -r requirements.txt
```

### 3. 数据准备与索引构建
1. 将原始邮件文本文件（`.txt`）放入 `data/ham/` 目录。
2. **关键步骤**：运行客户端工具构建加密索引。
   ```bash
   python client_se_index_builder.py
   ```
   > ⚠️ **注意**：此步骤会生成 `private_key.pem`并将加密后的索引上传至服务器，服务器只会存储哈希值。

### 4. 启动系统
1. 启动 Flask 服务器：
   ```bash
   python app.py
   ```
2. 在浏览器访问：`http://localhost:5000`

### 5. 使用流程
1. **加载私钥**：在网页顶部上传刚才生成的 `private_key.pem` 文件（私钥仅存于浏览器内存）。
2. **搜索邮件**：输入中文关键词（如“发票”、“合同”、“会议”）。
3. **解密查看**：点击搜索结果中的 "🔓 解密查看" 按钮。
4. **隐私验证**：观察控制台和网络请求，确认解密密钥和内容从未离开过您的浏览器。

## 🔐 安全模型

| 威胁场景 | 防护机制 |
|--------|----------------------|
| **服务器窃取邮件内容** | 邮件内容使用 AES-256-GCM 加密，服务器无私钥，仅存密文 |
| **搜索关键词泄露** | 使用可搜索加密（SE），服务器仅存储关键词的 SHA-256 哈希值 |
| **私钥泄露** | 私钥永远不传输、不存储于服务器，仅存在于用户浏览器内存中 |
| **中间人攻击** | 生产环境建议部署 HTTPS，防止传输过程被窃听 |
| **访问模式泄露** | 采用分块（Block）架构，结合 PIR 思想模糊具体文件访问请求 |
